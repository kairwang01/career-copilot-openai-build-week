### AI — `analyzeResume` (stubbed LLM, 30s, 5 VUs) — ⚠️ BLOCKED (pre-existing bug)

The harness, the LLM stub, auth, quota-disable, and credit seeding all work. The
AI calls were measured (latency was recorded) but returned non-200 because of a
**pre-existing runtime bug unrelated to SCRUM-43**:

```
deductCredits: all 3 attempts failed
  → admin.firestore.FieldValue.serverTimestamp() / admin.firestore.Timestamp.fromDate()
    is undefined at runtime with the current firebase-admin version
    (functions/src/admin/usageLog.ts, functions/src/handlers/onUserCreated.ts)
```

This crashes the credit-ledger write inside `deductCredits` (and also breaks the
`onUserCreated` provisioning trigger), so `analyzeResume` cannot return 200 in the
emulator regardless of the load harness. It is **not** introduced by this ticket —
the stub short-circuit lives in `resolveProvider`, which runs *after* the failing
credit transaction.

Latency that was captured before the failure (stubbed, overhead-only):

| Metric | Value (stub, errored at ledger write) |
|--------|----------------------------------------|
| avg | ~2.0 s |
| p90 | ~7.5 s |
| p95 | ~8.7 s |

These AI numbers are **not a valid baseline** — they reflect the failed retry/
back-off loop in `deductCredits` (3 attempts × exponential back-off), not real
endpoint behaviour. Once the firebase-admin bug is fixed, re-run
`SCENARIO=ai npm run loadtest:baseline` to capture the true overhead-only P95.

**Follow-up filed:** fix `admin.firestore.FieldValue` / `admin.firestore.Timestamp`
undefined-at-runtime (import `FieldValue`/`Timestamp` from `firebase-admin/firestore`).

### M3 caveat (real provider)

Even once unblocked, this AI P95 is the **overhead floor only** (stubbed LLM).
The real M3 number (community LLM router currently 30–90s; acceptable target
< 8s P95) is measured in **SCRUM-45** against the live provider.
